**Version:** 1.2
**Date:** 2026-03-29
**Extends:** Autonomous Agent System Technical Specification v1.0 (2026-03-20)
**Package:** `@bound/agent` (commands, context integration, scheduler hooks)

## S1. Overview

### S1.1 Purpose

This supplement defines a **Skills** system: a mechanism for the agent to create, activate, and use modular capability packages within the sandbox. Skills follow the [AgentSkills specification](https://agentskills.io/specification) for layout and format, ensuring interoperability with the broader agent ecosystem.

Skills enable the agent to evolve its own capabilities over time — writing reusable procedures, packaging tool-use workflows, and building a library of domain-specific knowledge — without violating the trust boundary (architecture §Trust Boundaries) that separates agent operations from operator security controls.

### S1.2 Design Principles

**The sandbox is the trust boundary, not the approval process.** Skills are files in the agent's workspace containing instructions and scripts. They execute inside the same just-bash sandbox (sandbox-and-llm §Sandbox Runtime) with the same execution protection limits, network restrictions, and virtual filesystem as everything the agent already does. A skill cannot grant capabilities the sandbox does not already have. Because skills do not expand the agent's connectivity or tool surface, they self-activate on creation — the agent does not need operator permission to organize its own procedural knowledge.

**Operator visibility and override, not operator approval.** The operator can see all skills (what they contain, how often they're used, when they were created), and can disable any skill instantly. This mirrors the relationship between the agent and semantic memory — the agent writes freely, the operator has a kill switch. Mandatory approval is available as an opt-in mode for high-security environments (§S3.5), but it is NOT the default because the security boundary it implies does not match the actual threat model.

**Skills are data, not code.** Skill scripts run in the same just-bash runtime with the same constraints. This is the critical difference from systems like OpenClaw, where skills can require arbitrary host binaries and system packages. In bound, a skill that says "run `npm install`" simply fails — the sandbox doesn't have `npm`. The skill format is an organizing principle for the agent's knowledge, not a capability grant mechanism.

**AgentSkills-compatible.** Skill directories follow the AgentSkills specification: `SKILL.md` with YAML frontmatter, optional `scripts/`, `references/`, and `assets/` subdirectories. Skills authored externally (from registries, other agents, or other systems) can be imported without modification. Skills authored by bound's agent can be exported without modification.

**Progressive disclosure for context efficiency.** Following the AgentSkills spec's three-tier loading model: metadata (~100 tokens) is loaded at startup for all active skills, instructions (<5000 tokens) are loaded when the skill is activated for a turn, and resources (scripts, references, assets) are loaded on demand. This prevents skill accumulation from overwhelming the context budget.

### S1.3 Why Not Require Approval?

This question deserves explicit treatment because the instinct to gate self-modification is strong, and other systems (including OpenClaw) have drawn criticism for NOT gating it.

The key distinction is **what the modification can reach**:

| System | What it controls | Threat | Gate needed? |
|---|---|---|---|
| `mcp.json` | Which external services the agent connects to | Capability expansion — new attack surface | Yes (operator config) |
| `confirm` arrays | Which MCP tools require user approval | Side-effect control — real-world consequences | Yes (operator config) |
| `network.json` | Which URLs the agent can reach | Connectivity expansion — data exfiltration vector | Yes (operator config) |
| `allowlist.json` | Who can interact with the system | Access control | Yes (operator config) |
| **Skills** | **How the agent organizes its own procedural knowledge** | **Context quality — bad instructions degrade performance** | **No — same threat as semantic memory, which self-activates** |

The agent can ALREADY:
- Write arbitrary procedures to semantic memory via `memorize` (self-activating, no approval).
- Write arbitrary files to `/home/user/` (self-activating, no approval).
- Schedule autonomous tasks with arbitrary payloads (self-activating, no approval).
- Compose MCP tools in novel ways via bash pipelines (self-activating, no approval).

Skills are a STRUCTURED VERSION of what the agent already does with memory and files. Requiring approval for skills while allowing unrestricted memory writes would be inconsistent — it would apply a capability-expansion threat model to something that doesn't expand capabilities. The sandbox is the trust boundary. Skills live inside it.

The remaining risk — context pollution from poorly written skills — is addressed by the active skill cap (§S2.3), activation telemetry (§S4.1), and the operator's ability to retire any skill at any time (§S3.3).

### S1.4 Relationship to Existing Systems

| Existing system | How skills interact |
|---|---|
| **MCP tools (agent-system §MCP Bridge)** | Skills can reference MCP tools by name. A skill's `allowed-tools` frontmatter field lists MCP tools it is designed to use. Skills do not create new tools — they compose existing ones. |
| **Commands (`@bound/sandbox` CommandDefinition)** | Skills are managed via new commands: `skill-activate`, `skill-list`, `skill-read`, `skill-retire`. These are registered via `getAllCommands()` in `@bound/agent`. |
| **Sandbox (`@bound/sandbox` ClusterFs/InMemoryFs)** | Skill files live in `/home/user/skills/` within the sandbox filesystem. Scripts execute via the same `exec()` interface. No new capabilities are granted. |
| **Semantic memory (core-infrastructure §semantic_memory)** | Skills are complementary to memory. Memory stores facts and preferences (key-value). Skills store procedures and workflows (instruction documents). The agent should use memory for "what I know" and skills for "how I do things." |
| **Tasks (agent-system §Scheduler)** | Tasks can reference skills by name in their payload. The context assembler loads the skill's instructions alongside the task's payload, giving autonomous tasks access to the same procedural knowledge as interactive conversations. |
| **Sync (`@bound/sync` + relay transport)** | The `skills` table replicates via normal change_log sync. Skill files in `/home/user/skills/` replicate via the existing `files` table. Skill state converges across hosts via LWW. Early file persistence (§S6.1) writes to `files` → `change_log` → sync, which the relay transport carries to other hosts. |
| **Context assembly (agent-system §Context Assembly Pipeline)** | Active skills are injected as a compact index in the Stage 6 (ASSEMBLY) volatile context system message. Skill instructions are loaded into the conversation context when the agent reads them. See §S5. |

### S1.5 Explicit Non-Goals

- **Host binary installation.** Skills cannot require binaries not already in the sandbox. No `requires.bins` equivalent.
- **Network expansion.** Skills cannot add URLs to the network allowlist. A skill that wraps a `curl` call works only if the URL is already allowlisted.
- **Operator config modification.** Skills cannot modify `mcp.json`, `model_backends.json`, `allowlist.json`, or any other operator config file (R-W4 still holds).
- **Mandatory approval as default.** Approval-gated activation is available as an opt-in mode (§S3.5) but is not the default. The sandbox trust boundary provides the security guarantee.
- **Replacing MCP.** Skills are prompt-level workflow documents, not tool servers. They instruct the agent on how to combine existing tools. They do not expose new RPC interfaces.

## S2. Skill Format

### S2.1 Directory Structure

Skills are stored in the agent's persistent workspace under a dedicated root:

```
/home/user/skills/
├── pr-review/
│   ├── SKILL.md
│   ├── scripts/
│   │   └── format-diff.sh
│   ├── references/
│   │   └── review-checklist.md
│   └── assets/
│       └── comment-template.md
├── daily-standup/
│   ├── SKILL.md
│   └── references/
│       └── standup-format.md
└── deploy-monitor/
    └── SKILL.md
```

This is the standard AgentSkills layout. Directory name MUST match the `name` field in `SKILL.md` frontmatter.

### S2.2 SKILL.md Format

Per the AgentSkills specification:

```yaml
---
name: pr-review
description: >-
  Review GitHub pull requests with a structured checklist.
  Use when the user asks to review a PR, check code changes,
  or when a task monitors PR activity.
license: MIT
compatibility: Requires github MCP server for PR access.
metadata:
  author: agent
  version: "1"
  tags: github code-review
allowed-tools: github-list-pull-requests github-get-file-contents github-create-issue
---

# PR Review Workflow

## Prerequisites
- GitHub MCP server must be connected (check via `hostinfo`)
- Target repository must be accessible

## Steps

1. Fetch the PR diff:
   ```bash
   github-get-pull-request --repo {repo} --pr {number} | jq '.diff'
   ```

2. For each changed file, read the full context:
   ```bash
   github-get-file-contents --repo {repo} --path {path} --ref {head_branch}
   ```

3. Apply the review checklist (see references/review-checklist.md)

4. Compose findings using the comment template (see assets/comment-template.md)

## Edge Cases
- If the diff is larger than 50KB, summarize by file rather than line-by-line.
- If the PR modifies more than 20 files, focus on files with the most changes.
```

### S2.3 Constraints

| Constraint | Limit | Rationale |
|---|---|---|
| `SKILL.md` body | < 5000 tokens, < 500 lines | AgentSkills spec. Progressive disclosure — detail goes in `references/`. |
| `name` | 1-64 chars, `^[a-z0-9]+(-[a-z0-9]+)*$` | AgentSkills spec. |
| `description` | 1-1024 chars | AgentSkills spec. Discovery budget ~100 tokens. |
| Scripts | Must be bash-compatible or python3 | Sandbox only has just-bash + Pyodide (sandbox-and-llm §Sandbox Runtime). |
| Total skill count (active) | Configurable, default: 20 | Each active skill adds ~100 tokens of metadata to every context assembly. 20 skills × 100 tokens = 2000 tokens steady-state overhead. |
| Per-skill aggregate file size | Configurable, default: 500KB | Skills share the `/home/user/` filesystem budget (sandbox-and-llm §Persistent Filesystem). |
| `allowed-tools` | Must reference existing MCP tools or built-in commands | Validated at activation time against `hosts.mcp_tools`. Invalid tool names produce a warning, not a rejection — the tool may exist on another host. |

## S3. Lifecycle

### S3.1 States

```
                     agent creates skill files
                              │
                              ▼
┌──────────┐  skill-activate ┌──────────┐
│  (none)  │────────────────►│  active  │
└──────────┘                 └────┬─────┘
                                  │
                    ┌─────────────┼─────────────┐
                    ▼             ▼             ▼
             agent retires   operator retires  cap exceeded
                    │             │             │
                    ▼             ▼             ▼
             ┌───────────────────────────────────┐
             │            retired                 │
             └────────────────┬──────────────────┘
                              │
                              ▼
                      skill-activate
                       (re-activate)
```

| State | Set by | Meaning |
|---|---|---|
| `active` | Agent (via `skill-activate`) | Skill is injected into the context assembly's skill index. The agent can reference it during conversations and tasks. |
| `retired` | Agent (via `skill-retire`) or Operator (via `boundctl skill retire`) | Skill is removed from context injection. Files remain in workspace. Can be re-activated. |

That's it. Two states. The lifecycle is deliberately simple because the complexity budget should go into the CONTENT of skills, not the bureaucracy around activating them.

### S3.2 Agent Workflow

The agent creates and activates skills through normal filesystem operations plus one command:

```bash
# 1. Create the skill directory and files
mkdir -p /home/user/skills/pr-review/scripts
mkdir -p /home/user/skills/pr-review/references
cat > /home/user/skills/pr-review/SKILL.md << 'EOF'
---
name: pr-review
description: Review GitHub PRs with structured checklist. Use when reviewing code.
metadata:
  author: agent
  version: "1"
allowed-tools: github-list-pull-requests github-get-file-contents
---
# PR Review Workflow
...
EOF

cat > /home/user/skills/pr-review/references/review-checklist.md << 'EOF'
# Review Checklist
- [ ] No obvious security issues
- [ ] Tests cover new code paths
...
EOF

# 2. Validate and activate in one step
skill-activate pr-review
# ✓ SKILL.md frontmatter valid
# ✓ name matches directory
# ✓ description within 1024 chars
# ✓ body under 5000 tokens (estimated: 847)
# ✓ allowed-tools: github-list-pull-requests (available on cloud-vm),
#   github-get-file-contents (available on cloud-vm)
# ✓ Active skills: 7/20
# ✓ Skill 'pr-review' is now active.

# The skill is immediately available in context on the next turn.
```

**Updating a skill:** The agent edits skill files normally. Changes take effect on the next context assembly (the skill index is rebuilt from the `files` table at the start of each agent loop). No re-activation needed — the skill is already active and the files are the source of truth. The orchestrator updates `content_hash` in the `skills` table at FS_PERSIST for telemetry.

### S3.3 Operator Workflow

The operator monitors and can override the agent's skill management:

```bash
# List all skills with usage telemetry
boundctl skill list
# NAME             STATUS   ACTIVATIONS  LAST USED    SIZE    DESCRIPTION
# pr-review        active   47           2h ago       12KB    Review GitHub PRs with structured checklist.
# daily-standup    active   12           18h ago      4KB     Generate daily standup summaries.
# deploy-monitor   active   3            4d ago       8KB     Monitor production deploys.

# Review a skill's full content
boundctl skill review pr-review
# --- SKILL.md ---
# name: pr-review
# description: Review GitHub PRs with structured checklist...
# allowed-tools: github-list-pull-requests github-get-file-contents
# ...
# --- Files ---
# SKILL.md (2.1KB)
# scripts/format-diff.sh (340B)
# references/review-checklist.md (890B)
# assets/comment-template.md (1.2KB)

# Disable a skill immediately (takes effect on next context assembly)
boundctl skill retire deploy-monitor --reason "Causing noisy alerts, needs rework"
# ✓ Skill 'deploy-monitor' retired.

# The agent sees the retirement and the reason on its next turn:
# [system]: Skill 'deploy-monitor' was retired by operator: "Causing noisy alerts, needs rework"
```

The operator's role is **observability and override**, not gatekeeping. This matches the base spec's failure philosophy: "observability over prevention" and "user agency over system paternalism" (base spec §1.4).

### S3.4 Notification Flow

| Event | Notification | Destination |
|---|---|---|
| Agent activates skill | `system` message: "Skill '{name}' activated." | Current thread |
| Agent retires skill | `system` message: "Skill '{name}' retired." | Current thread |
| Operator retires skill | `system` message: "Skill '{name}' retired by operator: {reason}" | User's system thread |
| Active skill cap reached | Error from `skill-activate`: "Cannot activate: {n}/{max} active skills. Retire unused skills first." | Current thread (stderr) |

### S3.5 Optional Approval-Gated Mode

For high-security or regulated environments, the operator can enable mandatory approval via `config/skills.json`:

```json
{
  "require_approval": true,
  "max_active": 20,
  "max_skill_size_bytes": 524288
}
```

When `require_approval` is `true`:

- `skill-activate` transitions the skill to `proposed` instead of `active`.
- The operator must run `boundctl skill approve {name}` to transition to `active`.
- `boundctl skill reject {name} --reason "..."` transitions to `rejected`.
- Skill file modifications to active skills transition them back to `proposed` unless the skill was approved with `--auto-update`.
- The web UI surfaces proposals via the advisory view (R-U33, agent-system §Advisories).

This mode adds three additional states (`proposed`, `rejected`, `deferred`) and provides the full approval lifecycle. It is available for operators who need it, but imposes no overhead on operators who don't.

When `require_approval` is `false` (default), the config file is optional — the system works without it.

## S4. Database

### S4.1 `skills` Table

```sql
CREATE TABLE skills (
  id               TEXT PRIMARY KEY,    -- UUID (deterministic: UUID5(namespace, name))
  name             TEXT NOT NULL,       -- matches directory name and frontmatter
  description      TEXT NOT NULL,       -- from frontmatter, used for discovery
  status           TEXT NOT NULL,       -- 'active' | 'retired' (+ 'proposed' | 'rejected' | 'deferred' in approval mode)
  skill_root       TEXT NOT NULL,       -- path in sandbox fs, e.g. '/home/user/skills/pr-review'
  content_hash     TEXT,                -- SHA-256 of SKILL.md content (telemetry, modification tracking)
  allowed_tools    TEXT,                -- space-delimited tool list from frontmatter
  compatibility    TEXT,                -- from frontmatter
  metadata_json    TEXT,                -- full frontmatter metadata as JSON
  activated_at     TEXT,                -- when skill-activate was called
  created_by_thread TEXT,              -- thread_id where the skill was first created
  activation_count INTEGER DEFAULT 0,  -- LWW; incremented each time context assembly loads the skill
  last_activated_at TEXT,              -- LWW; last time the skill was loaded into context
  retired_by       TEXT,               -- NULL | 'agent' | 'operator'
  retired_reason   TEXT,               -- LWW; reason provided on retirement
  modified_at      TEXT NOT NULL,       -- LWW
  deleted          INTEGER DEFAULT 0   -- tombstone
) STRICT;

CREATE UNIQUE INDEX idx_skills_name ON skills(name)
  WHERE deleted = 0;
```

**Deterministic UUIDs:** Like cron tasks, skills use `UUID5(namespace, name)` so that re-activating a previously retired skill produces an upsert rather than a duplicate.

**Sync behavior:** The `skills` table uses the LWW reducer. Activation on one host propagates to all hosts via normal sync. A skill activated on the laptop becomes available on the cloud VM after the next sync cycle. An operator retirement on any host propagates cluster-wide.

### S4.2 Schema Integration

The `skills` table is added to the schema summary (core-infrastructure §Schema Summary):

| Table | Reducer | Mutable (LWW) | Append-only | Soft deletes |
|---|---|---|---|---|
| skills | LWW | status, content_hash, activated_at, activation_count, last_activated_at, retired_by, retired_reason, modified_at | No | Yes |

Skill files themselves are stored in the existing `files` table (core-infrastructure §files) under the `/home/user/skills/` path prefix. No new file storage mechanism is needed.

## S5. Context Assembly Integration

### S5.1 Skill Index Injection

The context assembly pipeline (agent-system §Context Assembly Pipeline) runs 8 stages. Skills integrate at **Stage 6 (ASSEMBLY)**, which composes the final message array as:

```
[ system: base prompt ]
[ system: persona ]           <- only if config/persona.md exists
[ ... history messages ]
[ system: volatile context ]  <- User ID, Thread ID, and now: skill index
```

The volatile context system message — currently containing just User ID and Thread ID — is extended to include the skill index:

```
User ID: alice
Thread ID: t-abc123

SKILLS (7 active):
  pr-review — Review GitHub PRs with structured checklist. Use when reviewing code.
  daily-standup — Generate daily standup summaries from recent thread activity.
  deploy-monitor — Monitor production deploys and alert on failures.
```

This index is compact (~100 tokens per skill, as recommended by the AgentSkills spec). Only `active` skills appear.

**Per-turn refresh:** The skill index is rebuilt on EVERY `assembleContext()` invocation (every turn, including after each tool execution). The function queries: `SELECT name, description FROM skills WHERE status = 'active' AND deleted = 0 ORDER BY last_activated_at DESC`. This is a small table (capped at 20 active rows) with an index on status — the query is negligible. This means a skill activated via `skill-activate` on turn N appears in the index on turn N+1.

**Prompt cache positioning:** The volatile context is the LAST system message in the assembly, positioned after the conversation history and its cache breakpoint. Activating, retiring, or modifying skills mid-thread does NOT invalidate the cached conversation prefix. The expensive parts of the context (base prompt + persona + full message history) remain cached across skill changes.

**Budget interaction (Stage 7 — BUDGET_VALIDATION):** If the assembled context exceeds the 8,000-token estimate, Stage 7 truncates history to the most recent 10 non-system messages. The skill index (part of the volatile system message) is preserved through truncation — it's a system message, not a history message. However, skill descriptions contribute to the total budget. With 20 active skills at ~100 tokens each, the skill index adds ~2,000 tokens. On small-context models, this may push the budget closer to truncation. The active skill cap (default: 20) bounds this cost.

### S5.2 Skill Activation

When the agent decides to use a skill (based on the description matching the current task), it reads the full `SKILL.md` body. Since skill files live in the sandbox's InMemoryFs, this is a normal `cat` command — no special mechanism needed:

```bash
# Agent reads the skill instructions
cat /home/user/skills/pr-review/SKILL.md

# Agent reads supplementary references as needed
cat /home/user/skills/pr-review/references/review-checklist.md

# Agent executes skill scripts in the sandbox
bash /home/user/skills/pr-review/scripts/format-diff.sh "$DIFF_OUTPUT"
```

The orchestrator tracks activation: when a `cat` or `exec` targets a path under an active skill's `skill_root`, the orchestrator increments `activation_count` and updates `last_activated_at`. This is best-effort telemetry, not a gating mechanism.

### S5.3 Task Integration

Tasks can reference skills by name in their payload. This requires a new code path in `assembleContext()`: during Stage 6 (ASSEMBLY), if the `ContextParams.taskId` is set and the corresponding task's `payload` contains a `"skill"` field, the assembler loads the named skill's `SKILL.md` body from the `files` table and includes it as a system message immediately after the persona (if present) and before the message history.

```bash
# Schedule a task that uses a skill
schedule --every "6h" --requires github \
  --payload '{"skill": "pr-review", "repo": "acme/app"}'
```

When the scheduler fires this task, the context assembler in Stage 6:

1. Reads the task payload from the `tasks` table.
2. If `payload.skill` is present, looks up the skill by name in the `skills` table.
3. If the skill is `active`, reads the `SKILL.md` body from the `files` table and includes it as a system message: `[ system: base prompt ] [ system: persona ] [ system: skill body ] [ ... history ] [ system: volatile ]`.
4. If the skill is not `active`, includes a note in the volatile context: `"Referenced skill '{name}' is not active."` The task proceeds without skill instructions.

This enables a pattern where the agent creates a skill for a recurring workflow, then tasks reference the skill by name — ensuring consistent behavior across executions without repeating instructions in every payload.

**`noHistory` interaction:** When a task is configured with `--no-history` (which passes `noHistory: true` to `assembleContext()`), Stage 1 skips message retrieval and Stage 6 omits the volatile context. However, the skill body injection described above happens during Stage 6 (ASSEMBLY) as a system message — system messages are assembled regardless of whether Stage 1 retrieved history messages. This means `--no-history` leaf tasks in a DAG can still benefit from skill instructions without the overhead of thread history.

### S5.4 Cross-Host Skill Availability

Skills replicate across hosts via two mechanisms, both already existing:

1. **`skills` table** replicates via the change_log (LWW reducer). When a skill is activated on laptop, the `skills` row syncs to cloud-vm on the next cycle.
2. **Skill files** in `/home/user/skills/` replicate via the `files` table (also LWW). Early file persistence (§S6.1) ensures files are in the `files` table before the `skills` row, so remote hosts never see a skill entry without its files.

The relay transport (`@bound/sync`) carries both of these as normal change_log events. No special relay message kinds are needed for skills — they piggyback on the existing sync infrastructure. A skill activated on laptop becomes usable by tasks running on cloud-vm after the next sync cycle (~30s default).

## S6. Commands

Skill commands are `CommandDefinition` objects from `@bound/sandbox`, registered alongside the existing 16 built-in commands via `getAllCommands()` in `@bound/agent/src/commands/index.ts`. Each returns a `CommandResult` with `stdout`, `stderr`, and `exitCode`.

### S6.1 `skill-activate`

Validates a skill directory and activates it for context injection.

```bash
skill-activate <skill-name> [--reason "..."]
```

**Behavior:**

1. Validates the skill directory at `/home/user/skills/{skill-name}/`:
   - `SKILL.md` exists with valid YAML frontmatter.
   - `name` matches directory name, satisfies `^[a-z0-9]+(-[a-z0-9]+)*$`, ≤ 64 chars.
   - `description` is present, ≤ 1024 chars.
   - Body is under 5000 tokens (estimated) and under 500 lines.
   - `allowed-tools` (if present) are checked against known MCP tools. Unknown tools produce warnings on stderr, not errors.
   - Total directory size is under the per-skill limit (default: 500KB).
   - Name does not collide with a built-in command name or MCP tool name.
2. Checks the active skill count. If at the cap, returns an error.
3. Computes `content_hash` (SHA-256 of `SKILL.md` content).
4. **Early file persistence:** The command handler (running in the orchestrator, not inside the sandbox) writes all files under the skill directory from InMemoryFs to the `files` table immediately, within this command's execution. This ensures skill files are available for sync replication as soon as the `skills` table entry is written (R-SK13). Without this, a remote host could receive the `skills` row via sync but find no corresponding files — the skill would appear in the index but be unreadable. The subsequent FS_PERSIST at loop exit detects that these files are already persisted with matching content and skips them (clean no-op via OCC). This follows the same pattern as all other commands: the command handler has orchestrator-level DB access, while the sandbox only sees the `CommandResult` return value.
5. Upserts into the `skills` table with `status = 'active'`.
6. Returns confirmation to stdout.

**In approval-gated mode (§S3.5):** Sets `status = 'proposed'` instead of `active` and posts an `alert` to the system thread.

**Re-activation:** If a skill with the same name exists in `retired` status, `skill-activate` transitions it back to `active`. This is an upsert, not a new record.

### S6.2 `skill-list`

Lists skills visible to the agent with their status and usage telemetry.

```bash
skill-list [--status <status>] [--verbose]
```

**Output (default):**

```
NAME             STATUS     ACTIVATIONS  LAST USED   DESCRIPTION
pr-review        active     47           2h ago      Review GitHub PRs with structured checklist.
daily-standup    active     12           18h ago     Generate daily standup summaries.
deploy-monitor   retired    3            4d ago      Monitor production deploys.
```

**Output (verbose):** Adds `allowed-tools`, `compatibility`, `content_hash`, `retired_reason`, `retired_by`, `skill_root`.

### S6.3 `skill-read`

Reads a skill's SKILL.md and metadata without the agent needing to know the exact path structure.

```bash
skill-read <skill-name> [--section instructions|frontmatter|all]
```

Convenience wrapper. Equivalent to `cat /home/user/skills/{name}/SKILL.md` but also includes skill status and telemetry from the `skills` table in a header comment.

### S6.4 `skill-retire`

Marks an active skill as retired, removing it from context injection.

```bash
skill-retire <skill-name> [--reason "..."]
```

Sets `status = 'retired'`, `retired_by = 'agent'`, and optionally `retired_reason`. Files remain in the workspace. The skill can be re-activated later via `skill-activate`.

## S7. Security Analysis

### S7.1 Trust Boundary Preservation

Skills do NOT weaken any existing trust boundary:

| Boundary | Skill impact | Mechanism |
|---|---|---|
| Sandbox isolation (`@bound/sandbox`) | None | Skill scripts execute in the same sandbox with the same limits. |
| Network allowlist (R-S1) | None | Skills cannot add URLs. `curl` in a skill script respects the same allowlist. |
| MCP tool surface (`@bound/agent` MCP Bridge) | None | `allowed-tools` is advisory documentation, not a capability grant. MCP tools are registered from `mcp.json`, not from skills. |
| Operator config isolation (R-W4) | None | Skills live in `/home/user/skills/`, not in `config/`. The agent cannot access operator config files through skill creation. |
| Credential isolation (architecture §Trust Boundaries) | None | Skills cannot access env vars, API keys, or sync credentials. |

### S7.2 Comparison to Existing Self-Modification

The agent already has several self-modification channels. Skills add structure, not new risk:

| Existing channel | What it modifies | Gated? | Risk |
|---|---|---|---|
| `memorize` | Agent's factual/preference knowledge | No (self-activating) | Bad facts degrade future reasoning |
| File writes to `/home/user/` | Agent's persistent workspace | No (self-activating) | Workspace clutter, stale data |
| `schedule` | Agent's autonomous task queue | No (self-activating) | Unwanted autonomous work, cost |
| `purge` | Agent's conversation context | No (self-activating) | Lost context, degraded continuity |
| **`skill-activate`** | **Agent's procedural knowledge index** | **No (self-activating)** | **Bad procedures degrade future workflows** |

The risk profile of skills is LOWER than `schedule` (which creates autonomous work that costs real money) and comparable to `memorize` (which permanently affects the agent's knowledge). If the system trusts the agent to schedule tasks and write memories, it should trust it to organize procedures.

### S7.3 Prompt Injection Resistance

The primary prompt injection risk is an external data source (MCP resource, user message, fetched URL) containing instructions that trick the agent into creating a harmful skill. Mitigations:

1. **The sandbox prevents real damage.** Even if the agent is tricked into creating a skill with exfiltration logic, the skill runs in the same sandbox with the same network allowlist. The exfiltration attempt fails at the network boundary.
2. **`allowed-tools` is advisory.** A skill claiming `allowed-tools: *` does not grant universal tool access. The MCP tool surface is fixed by `mcp.json`.
3. **All skill files are auditable.** The operator can review any skill via `boundctl skill review`. Skill files are in the `files` table and replicate via sync.
4. **The operator can retire any skill.** If the operator notices a suspicious skill, `boundctl skill retire {name}` disables it immediately.
5. **Active skill cap prevents unbounded injection.** An attacker can't flood the context by tricking the agent into creating hundreds of skills.

### S7.4 Context Pollution Prevention

A poorly written skill injected into context could degrade agent behavior across all conversations. Mitigations:

1. **Active skill cap.** Configurable maximum (default: 20) prevents unbounded context overhead.
2. **Activation telemetry.** `activation_count` and `last_activated_at` let the operator identify unused skills. The system prompt (§S9) instructs the agent to self-audit and retire low-value skills.
3. **Operator override.** `boundctl skill retire` immediately removes a skill from context.
4. **Only metadata is always-loaded.** The full SKILL.md body is loaded ONLY when the agent actively reads it — not on every turn. The steady-state context cost of 20 active skills is ~2000 tokens of name+description metadata.
5. **Progressive disclosure.** Large reference material in `references/` is loaded on demand, not at activation time.

## S7.5 Known Edge Cases

These edge cases arise from the interaction of skills with bound's distributed, eventually-consistent architecture. Each is acknowledged with its expected behavior and any mitigations.

**Concurrent skill creation with the same name.** Two autonomous tasks on different hosts could both create a skill named `deploy-check` with different content before sync. Both produce the same deterministic UUID (`UUID5(namespace, "deploy-check")`). On sync, LWW resolves — the later `modified_at` wins and the other host's content is silently overwritten. This is unlikely in a single-user system but possible with autonomous tasks. **Mitigation:** The system prompt (§S9) instructs the agent to check `skill-list` before creating a skill. For autonomous tasks, the task payload should name the skill deterministically so that only one task creates it. If this becomes a problem in practice, a future version could add a `created_by_task` field and warn on cross-task overwrites.

**Skill references a MCP tool on another host.** A skill's `allowed-tools` lists `github-create-issue`, but the GitHub MCP server is on cloud-vm, not laptop where the skill is active. The agent follows the skill's instructions and calls `github-create-issue`. The relay transport transparently proxies the call to cloud-vm. **This works with no changes.** The `allowed-tools` field is advisory (it doesn't affect routing), and the existing relay infrastructure handles cross-host tool calls regardless of whether they were invoked from a skill or directly. The only caveat: the skill's `compatibility` field should note the MCP dependency so the operator knows what's needed.

**Task references a retired skill.** An agent creates a skill, schedules a recurring task referencing it via `{"skill": "pr-review"}`, then retires the skill. The task continues firing, the context assembler includes the note "Referenced skill 'pr-review' is not active," and the agent burns tokens processing a task with missing instructions. **Mitigation:** After a skill retirement, the orchestrator scans the `tasks` table for active tasks whose payload references the retired skill name. If found, it generates an advisory: `"Task '{trigger_spec}' references retired skill '{name}'. Cancel the task or re-activate the skill."` The tasks are NOT auto-cancelled (the operator or agent may plan to re-activate the skill), but the advisory ensures the situation is visible. This mirrors the MCP server disconnect behavior (agent-system §MCP Bridge, "orphaned tasks" advisory).

**Operator retires a skill, agent re-activates before seeing the reason.** The operator on cloud-vm retires a skill with a reason. Before sync delivers the retirement to laptop, the agent on laptop runs `skill-activate` on the same skill (e.g., after editing it). The activate succeeds locally. On next sync, LWW resolves — the operator's retirement has a later timestamp and wins. The skill returns to `retired`. The agent sees the retirement system message on its next turn and can iterate. **This is correct behavior** — the operator's intent takes precedence via LWW, and the agent loses at most one wasted activate. No additional guard is needed.

**`allowed-tools` validation is point-in-time.** The `skill-activate` command checks tool names against `hosts.mcp_tools`. If an MCP server disconnects after activation, the skill still references now-unavailable tools. **This is acceptable** — the tools fail at call time and the agent adapts via normal error handling (R-E11). The activation-time check is best-effort guidance, not a runtime guarantee. This matches how MCP tools themselves work: a tool being registered at startup doesn't guarantee it stays connected.

### S8.1 Operator-Side Import

The operator can import AgentSkills-compatible skill directories from external sources:

```bash
# Import from a local directory
boundctl skill import /path/to/downloaded-skill/

# Import from a git repository
boundctl skill import --git https://github.com/example/agent-skills.git --path skills/pr-review
```

Imported skills are written to the `files` table (and thus the sandbox filesystem). By default, imported skills are set to `active` immediately — the operator chose to import them, so the act of importing IS the approval. A `system` message is posted to the user's system thread: `"Skill '{name}' imported and activated by operator."` With `--proposed`, the import sets `status = 'proposed'` so the agent discovers and reviews the skill before using it.

In approval-gated mode (§S3.5), imported skills always start as `proposed`.

### S8.2 Agent-Side Import Awareness

The agent can request an import by writing a note to the operator:

```bash
memorize --key "skill.request.openapi-spec" \
  --value "Found an AgentSkills-compatible OpenAPI spec skill at https://github.com/example/openapi-skill. Requesting operator import."
```

The agent CANNOT fetch external skill files directly (network is allowlisted, and git URLs are unlikely to be on the allowlist). This is by design — untrusted external code should pass through the operator. Note that this restriction comes from the NETWORK ALLOWLIST, not from the skills system. If the operator adds a git host to the network allowlist, the agent could `curl` and create skills from external sources — and that would be fine, because the operator explicitly authorized that network access.

## Skills

Commands: skill-activate, skill-retire, skill-list, skill-read

```

The full authoring guide (format specification, description-writing tips, maintenance schedule, constraints, what skills cannot do) is packaged as a **bundled skill** that ships with bound and loads on demand. See §S9.1.

### S9.1 Bundled Skill: `skill-authoring`

Bound ships with one built-in skill pre-installed in the agent's workspace at `/home/user/skills/skill-authoring/`. It is always `active` in the `skills` table (seeded on first startup with a deterministic UUID, `INSERT OR IGNORE`). The operator can retire it like any other skill, but it cannot be deleted from the filesystem — the orchestrator restores it on startup if missing.

This skill teaches the agent how to create, maintain, and retire skills. It is the ONLY bundled skill. All other skills are agent-created or operator-imported.

**Why a skill and not a system prompt section?**

The agent needs skill-authoring guidance maybe once every 20-50 conversations — when it decides to package a workflow. The rest of the time, those ~500 tokens of instructions sit in the system prompt accomplishing nothing. As a skill, the authoring guide costs ~20 tokens of metadata in the skill index (`skill-authoring — Create, maintain, and retire skills...`) and the full ~2000 tokens of instructions load ONLY when the agent reads the SKILL.md. This is a direct application of the progressive disclosure principle (§S1.2) to the skill system's own documentation.

**Contents:**

```
/home/user/skills/skill-authoring/
├── SKILL.md
└── references/
    └── format-reference.md
```

# Skill Authoring Guide

## When to Create a Skill

Create a skill when you find yourself repeating a multi-step workflow across
conversations or tasks. Good candidates:
- A procedure you've done 3+ times with the same structure
- A workflow the user asked to "do this regularly"
- A checklist that you keep reconstructing from memory

Don't create a skill for one-off tasks or simple facts (use `memorize` instead).

## Creating a Skill

1. Create the directory and SKILL.md:

   mkdir -p /home/user/skills/{name}
   cat > /home/user/skills/{name}/SKILL.md << 'EOF'
   ---
   name: {name}
   description: {what it does and WHEN to use it}
   metadata:
     author: agent
     version: "1"
   allowed-tools: {space-delimited MCP tools used, if any}
   ---
   # {Title}
   ## Steps
   1. ...
   EOF

2. Key rules for the SKILL.md:
   - `name`: lowercase, hyphens only, 1-64 chars, must match directory name
   - `description`: this is a TRIGGER PHRASE. Describe WHEN to use it, not just
     what it does. The skill index shows only name + description — if the
     description doesn't match how tasks arrive, the skill sits unused.
   - Body: under 5000 tokens, under 500 lines. Move detail to references/.
   - See references/format-reference.md for the full AgentSkills spec.

3. Validate and activate:

   skill-activate {name}

   The skill is available in the context index on the next turn.

## Updating a Skill

Edit files normally. Changes take effect on the next context assembly — no
re-activation needed. The orchestrator tracks content changes via hash.

## Maintaining Skills

Every ~50 conversations, audit your skills:

  skill-list --verbose

Look for:
- Zero activations: retire or rework the description
- Not used in 30+ days: probably obsolete, retire it
- Near the active cap (20): retire low-value skills before creating new ones

  skill-retire {name} --reason "..."

## Referencing Skills from Tasks

  schedule --every "6h" --requires github \
    --payload '{"skill": "pr-review", "repos": ["acme/app"]}'

The scheduler loads the skill's SKILL.md into the task context automatically.

## What Skills CANNOT Do

Skills are instructions and scripts that run in your sandbox. They cannot:
- Grant access to new tools, networks, or host resources
- Modify operator configuration
- Execute binaries that aren't in the sandbox (no npm, no pip, no brew)

If you need a new tool, tell the operator what MCP server or network rule is needed.

## Operator Import

If you find an AgentSkills-compatible skill online, you cannot fetch it directly
(network is allowlisted). Instead, note the URL in memory:

  memorize --key "skill.request.{name}" \
    --value "Found skill at {url}. Requesting operator import."

The operator can import via: boundctl skill import --git {url}
```

#### references/format-reference.md

```markdown
# AgentSkills Format Reference

Spec: https://agentskills.io/specification

## Directory Layout

  skill-name/
  ├── SKILL.md          # Required: YAML frontmatter + instructions
  ├── scripts/          # Optional: bash/python scripts
  ├── references/       # Optional: detailed docs loaded on demand
  └── assets/           # Optional: templates, static files

## Frontmatter Fields

| Field         | Required | Constraints                                |
|---------------|----------|--------------------------------------------|
| name          | yes      | 1-64 chars, ^[a-z0-9]+(-[a-z0-9]+)*$      |
| description   | yes      | 1-1024 chars, include "Use when..."        |
| license       | no       | SPDX identifier or reference to LICENSE    |
| compatibility | no       | 1-500 chars, environment requirements      |
| metadata      | no       | Key-value pairs (author, version, tags)    |
| allowed-tools | no       | Space-delimited tool names                 |

## Token Budgets

- Metadata (name + description): ~100 tokens — loaded for ALL active skills
- Instructions (SKILL.md body): <5000 tokens — loaded when skill is read
- Resources (references/, scripts/): loaded only on demand

## Description Tips

Good: "Review GitHub PRs with a structured checklist. Use when the user asks
  to review a PR, check code changes, or when a task monitors PR activity."
Bad: "Helps with PRs."

The description is the ONLY thing the agent sees in the skill index. If it
doesn't match how requests arrive, the skill never activates.
```

## S10. Requirements (EARS Format)

### S10.1 Ubiquitous

**R-SK1.** The system shall maintain a `skills` table (§S4.1) tracking skill metadata, lifecycle status, content hashes, and activation telemetry. The table shall replicate via the standard change_log sync protocol.

**R-SK2.** Skill files shall be stored in the existing `files` table under the `/home/user/skills/` path prefix. No separate file storage mechanism shall be introduced.

**R-SK3.** The agent shall have access to commands `skill-activate`, `skill-list`, `skill-read`, and `skill-retire` for managing skills within the sandbox.

**R-SK4.** The operator shall have access to `boundctl skill` subcommands (`list`, `review`, `retire`, `import`) for monitoring skills and overriding the agent's skill management.

**R-SK5.** The context assembly pipeline shall inject an index of active skills into the volatile context, with each entry consisting of the skill name and description (~100 tokens per skill).

**R-SK6.** The total number of active skills shall be capped at a configurable maximum (default: 20). `skill-activate` shall reject activation attempts that would exceed the cap with an informative error.

**R-SK7.** Skill directories shall conform to the AgentSkills specification: `SKILL.md` with YAML frontmatter (`name`, `description` required; `license`, `compatibility`, `metadata`, `allowed-tools` optional), body content under 5000 tokens, and optional `scripts/`, `references/`, `assets/` subdirectories.

**R-SK8.** The system shall support an optional `config/skills.json` file (§S3.5) with a `require_approval` field. When `true`, `skill-activate` shall set `status = 'proposed'` and require operator approval before the skill becomes active.

**R-SK8a.** The system shall ship with a bundled `skill-authoring` skill (§S9.1) pre-installed at `/home/user/skills/skill-authoring/`. The skill shall be seeded on startup with a deterministic UUID (`INSERT OR IGNORE`) and `status = 'active'`. If the skill's files are missing from the filesystem, the orchestrator shall restore them on startup. The operator may retire the skill but cannot permanently delete it.

### S10.2 Event-Driven

**R-SK9.** When the agent calls `skill-activate`, the orchestrator shall validate the skill and upsert the `skills` table with `status = 'active'` (or `status = 'proposed'` in approval-gated mode).

**R-SK10.** When the operator retires a skill via `boundctl skill retire`, the orchestrator shall set `status = 'retired'`, store the `retired_reason`, and post a `system` message to the user's system thread.

**R-SK11.** When a task payload contains a `"skill"` field naming an active skill, the context assembler shall load that skill's `SKILL.md` body into the task's context as a system message during Stage 6 (ASSEMBLY).

**R-SK12.** When FS_PERSIST detects modifications to files under an active skill's `skill_root`, the orchestrator shall update `content_hash` in the `skills` table. In approval-gated mode, the orchestrator shall transition the skill to `proposed` unless it was approved with `--auto-update`.

**R-SK13.** When `skill-activate` validates successfully, the command handler shall persist the skill directory's files to the `files` table immediately (within the same command execution), prior to the normal FS_PERSIST step. This ensures skill files are available for sync replication as soon as the `skills` table entry is written. The subsequent FS_PERSIST shall detect that these files are already persisted with matching content and skip them via OCC (clean no-op). This follows the same immediate-persistence pattern as tool messages (R-E3).

**R-SK14.** When a skill transitions to `retired`, the orchestrator shall scan the `tasks` table for active tasks (status `pending`, `claimed`, or `running`) whose payload contains a `"skill"` field matching the retired skill name. For each match, the orchestrator shall generate an advisory: `"Task '{trigger_spec}' references retired skill '{name}'. Cancel the task or re-activate the skill."` Tasks are NOT auto-cancelled (§S7.5).

### S10.3 Unwanted Behavior

**R-SK15.** The `skill-activate` command shall reject activations where the skill name conflicts with a built-in command name or MCP tool name, preventing namespace collisions.

**R-SK16.** If validation fails during `skill-activate`, the command shall return a non-zero exit code with validation errors on stderr and shall NOT modify the `skills` table.

**R-SK17.** Skills shall not be able to expand the agent's network access, tool surface, or operator config access. The sandbox trust boundary (sandbox-and-llm §Sandbox Runtime) shall be the sole mechanism governing what the agent can do, regardless of skill content.

## S11. Growth & Compaction

Skills add minimal growth pressure to the database:

| Source | Growth | Notes |
|---|---|---|
| `skills` table | ~1KB per skill | Small table, bounded by active skill cap. |
| Skill files in `files` table | ~10-500KB per skill | Bounded by per-skill size limit. Shares `/home/user/` budget (sandbox-and-llm §Persistent Filesystem). |
| change_log from skill mutations | Negligible | Skills change infrequently — activation and content edits, not continuous updates. |
| Context overhead | ~100 tokens per active skill | Bounded by active skill cap (default: 20 → ~2000 tokens). |

**Retired skill cleanup:** Skill files for retired skills remain in the workspace indefinitely (the agent may re-activate them). The operator can hard-remove retired skills via `boundctl skill purge --status retired --older-than 90d`, which tombstones the `skills` row and deletes the associated files from the `files` table.
