## Pre-requisites

- Two or more bound instances configured in a hub-spoke topology
- Shared keyring with Ed25519 keypairs for all instances
- At least one MCP server configured on a spoke (e.g., a simple echo tool)
- `sync.json` configured with hub URL and optional `relay` subsection
- Network connectivity between hub and spokes

## Test Scenarios

### 1. Cross-Host Tool Call (AC1.1)

**Setup:** Hub H, Spoke A (requester), Spoke B (has MCP tool "echo-server-echo")

**Steps:**
1. Start all three instances
2. Wait for initial sync to complete (all hosts registered)
3. On Spoke A, run: `commands` — verify Spoke B's tool appears under "REMOTE (via relay)"
4. On Spoke A, invoke the remote tool via the agent (e.g., ask the agent to use `echo-server-echo`)
5. Observe the agent enters RELAY_WAIT state
6. Wait for result to return

**Expected:**
- Tool result matches what Spoke B would return locally
- No errors in any instance's logs
- `relay_cycles` table on Spoke A shows outbound entry
- `turns` table on Spoke A shows `relay_target` = Spoke B's hostname, `relay_latency_ms` > 0

### 2. Eager Push Delivery (AC2.1, AC2.2)

**Setup:** Hub H, Spoke A (addressable, has `sync_url`)

**Steps:**
1. On Spoke A, write a relay outbox entry targeting Hub H
2. Spoke A syncs with Hub H
3. Hub H processes and sends result
4. Check Hub H's logs for eager push attempt

**Expected:**
- Hub H attempts HTTP POST to Spoke A's `/api/relay-deliver`
- If Spoke A is reachable, delivery is near-instant (sub-second)
- If Spoke A receives same message via both eager push AND sync, only one inbox entry exists (dedup)

### 3. NAT'd Host Polling (AC1.8)

**Setup:** Hub H, NAT'd Spoke (no `sync_url` in hosts table)

**Steps:**
1. Configure a spoke without `sync_url` (NAT'd/firewalled)
2. Send a tool call targeting the NAT'd spoke
3. Observe that eager push is skipped (no sync_url)
4. NAT'd spoke polls via regular sync interval
5. Eventually receives the tool call and responds

**Expected:**
- Delivery latency is bounded by sync interval (not immediate)
- No eager push attempts in Hub logs for this spoke
- Tool call completes successfully via polling

### 4. Reachability Degradation (AC2.3, AC2.4)

**Setup:** Hub H, Spoke A (initially reachable)

**Steps:**
1. Stop Spoke A's web server (simulate unreachable)
2. Send 3 relay messages targeting Spoke A via another spoke
3. Check Hub H's reachability tracker state
4. Restart Spoke A
5. Spoke A syncs successfully
6. Send another relay message

**Expected:**
- After 3 failed pushes, Hub logs "Skipping eager push to unreachable spoke"
- After Spoke A syncs, reachability resets
- Subsequent messages attempt eager push again

### 5. Cancel Propagation (AC7.1, AC7.2)

**Setup:** Hub H, Spoke A (requester), Spoke B (slow MCP tool)

**Steps:**
1. Spoke A sends tool call to Spoke B (use a tool with artificial delay)
2. While RELAY_WAIT is active, cancel the operation (Ctrl+C or agent cancel)
3. Check relay outbox on Spoke A

**Expected:**
- Cancel outbox entry written with `kind = "cancel"` and `ref_id` pointing to original request
- `sync:trigger` emitted with reason "relay-cancel"
- Agent exits RELAY_WAIT state promptly
- Spoke B (if it hasn't started execution) discards the request

### 6. Hub Migration Drain (AC4.1-AC4.5)

**Setup:** Hub A (current), Hub C (new), Spoke B

**Steps:**
1. Spoke B has active relay traffic with Hub A
2. Run `boundctl set-hub <hub-c-url>` on Hub A
3. Observe drain progress messages in CLI output
4. After drain completes (or timeout), verify hub switch

**Expected:**
- `relay_draining` flag set in `host_meta`
- Spoke B holds back request-kind entries during drain
- Response-kind and cancel entries continue flowing
- After hub switch, Spoke B's held entries deliver to Hub C on first sync
- If drain times out (120s default), switch proceeds anyway with warning

### 7. Payload Size Enforcement (AC9.1)

**Steps:**
1. Attempt to write a relay outbox entry with payload > 2MB
2. Attempt to insert a relay inbox entry with payload > 2MB

**Expected:**
- Both operations throw `PayloadTooLargeError`
- Payloads under 2MB succeed normally

### 8. Pruning (AC9.3, AC8.4)

**Steps:**
1. Let relay traffic flow for some time
2. Mark some outbox entries as delivered and inbox entries as processed
3. Wait for pruning interval (or trigger manually)
4. Check relay tables

**Expected:**
- Delivered outbox entries older than 5 minutes are hard-deleted
- Processed inbox entries older than 5 minutes are hard-deleted
- `relay_cycles` entries older than 30 days are pruned
- Non-delivered/non-processed entries are NOT pruned regardless of age

### 9. Proxy Endpoints Removed (AC3.1, AC3.2)

**Steps:**
1. Start a bound instance
2. HTTP POST to `http://localhost:<port>/api/mcp-proxy`
3. HTTP POST to `http://localhost:<port>/api/file-fetch`

**Expected:**
- Both endpoints return 404
- No proxy-related code exists in the codebase

### 10. Commands Display Tiers (AC8.5)

**Steps:**
1. Start an instance with both local MCP tools and remote hosts advertising tools
2. Run the `commands` command

**Expected:**
- Output shows three sections: "Built-in:", "LOCAL (MCP):", "REMOTE (via relay):"
- Local MCP tools appear under LOCAL
- Remote tools appear under REMOTE with `[host: <hostname>]` attribution
- Built-in commands (commands, hostinfo, etc.) appear under Built-in

### 11. Idempotency (AC5.1, AC5.2, AC5.3)

**Steps:**
1. Send the same relay request twice with identical `idempotency_key`
2. Check execution count on target

**Expected:**
- Tool executes only once (first request)
- Second request returns cached result without re-execution
- After 5 minutes, cache expires and re-execution is allowed
