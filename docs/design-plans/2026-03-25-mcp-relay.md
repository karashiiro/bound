# MCP Relay Transport Design

## Summary

This design replaces Bound's direct HTTP proxy mechanism for cross-host MCP operations with a hub-mediated relay transport embedded in the sync protocol. Currently, hosts behind NAT cannot participate in distributed MCP workflows because they lack directly addressable HTTP endpoints. The relay transport solves this by introducing two ephemeral local-only tables (`relay_outbox` and `relay_inbox`) that store request/response messages. These messages travel through the hub during a new fourth sync phase (RELAY), enabling spoke-to-spoke communication without requiring direct addressability. The hub acts as the sole relay hop: Spoke A writes a request to its outbox, the hub delivers it to Spoke B's inbox during sync, Spoke B executes the MCP call locally and writes the response to its outbox, and the hub returns that response to Spoke A's inbox on its next sync cycle.

For latency optimization, the hub attempts eager HTTP push to addressable hosts via a new `/api/relay-deliver` endpoint, falling back to sync-only delivery on failure. The agent loop gains a RELAY_WAIT sub-state that writes relay requests, triggers immediate sync, and polls for responses — transparently handling remote tool calls as if they were local. Idempotency prevents duplicate executions on retry, and a drain protocol ensures safe hub migration by holding back new requests while in-flight messages complete. This design removes all direct HTTP proxy code (`/api/mcp-proxy`, `/api/file-fetch`, `proxyToolCall()`) and provides the foundation for future relay-based features like inference routing and cross-host event propagation.

## Definition of Done

Replace the base spec's direct HTTP proxy mechanism (`/api/mcp-proxy`, `/api/file-fetch`, `proxyToolCall()`) with a hub relay transport layer embedded in the sync protocol. All MCP request/response operations — tool calls, resource reads, prompt invocations, and cache warming — flow through the relay via the hub, removing the requirement that target hosts be directly addressable.

**Success criteria:**

- A host behind NAT can have its MCP tools called by any other host in the cluster, with the hub relaying the request/response
- Addressable hosts get eager-pushed relay messages for lower latency (~3s vs ~7.5s polling)
- The old `/api/mcp-proxy` and `/api/file-fetch` endpoints are deleted; `proxyToolCall()` is replaced with relay routing
- Hub migration safely drains in-flight relay messages before switching
- Idempotency prevents duplicate tool executions on retry
- The agent loop's RELAY_WAIT sub-state integrates transparently (agent doesn't know if a tool is local or remote)
- Cancel propagates to abort waiting on remote tool results
- Metrics track relay usage (`relay_target`, `relay_latency_ms` on `turns`; `relay_cycles` table)

**Key exclusions:** No streaming/inference routing, no platform connectors, no cross-host event propagation (not E2E testable yet), no activity status forwarding (only relevant for delegated loops), no web UI loop delegation (requires inference routing).

## Glossary

- **Hub**: The designated central host in a Bound cluster that coordinates sync operations and routes relay messages between spokes.
- **Spoke**: Any non-hub host in the cluster that syncs with the hub and can send/receive relay messages through it.
- **Relay transport**: A store-and-forward messaging system for delivering MCP requests and responses between hosts via the hub, eliminating the need for direct peer-to-peer connectivity.
- **Store-and-forward**: A messaging pattern where messages are written to local storage (outbox), transmitted asynchronously during sync, and delivered to the recipient's local storage (inbox) for processing.
- **Eager push**: An optimization where the hub immediately POSTs relay messages to addressable spokes via HTTP rather than waiting for the next sync cycle, reducing latency from ~7.5s (polling) to ~3s.
- **RELAY_WAIT**: A sub-state within the agent loop's TOOL_EXECUTE phase where the agent writes a relay request to the outbox, triggers immediate sync, and polls the inbox for a matching response.
- **Idempotency key**: A deterministic hash derived from the request's kind, tool/resource name, arguments, and rounded timestamp, used to deduplicate requests and prevent duplicate execution on retry.
- **Site ID**: A 16-byte hex identifier derived from the SHA-256 hash of a host's Ed25519 public key (SPKI format), uniquely identifying each host in the cluster.
- **MCP (Model Context Protocol)**: A protocol for exposing tools, resources, and prompts from external services to LLM agents. Bound bridges MCP servers into its agent loop.
- **Tool routing**: The process of selecting which host should execute a requested MCP tool, based on which hosts advertise the tool in `hosts.mcp_tools` and their sync recency.
- **Host failover**: When a relay request times out, the agent attempts the same tool call on the next eligible host until all hosts are exhausted.
- **Relay drain**: A protocol phase during hub migration where the old hub signals spokes to hold back new requests while in-flight responses complete.
- **Request-kind / Response-kind**: Classification of relay message types. Request-kind messages initiate operations (`tool_call`, `resource_read`, `prompt_invoke`, `cache_warm`, `cancel`). Response-kind messages return results (`result`, `error`).
- **ref_id**: A field in relay messages linking responses and cancels back to the original request's ID.
- **Quiescence**: A state where the cluster reduces sync frequency after prolonged user inactivity, trading relay latency for reduced resource usage.
- **Change-log outbox pattern**: Bound's data replication pattern: writes to synced tables insert both the row and a change_log entry in a transaction; sync delivers these to the hub for distribution.

## Architecture

Store-and-forward relay embedded in the sync protocol. Two local-only tables (`relay_outbox`, `relay_inbox`) hold ephemeral messages that travel between hosts via the hub during a new 4th sync phase (RELAY). The hub is the only relay hop — spoke-to-spoke messages always travel Spoke A → Hub → Spoke B. For addressable hosts, the hub eager-pushes messages via HTTP POST for lower latency.

### Relay Tables

Non-replicated, NOT synced via change_log. Hard-deleted after short retention (5 minutes).

**`relay_outbox`** — messages this host wants to send:

```sql
CREATE TABLE relay_outbox (
  id              TEXT PRIMARY KEY,
  target_site_id  TEXT NOT NULL,
  kind            TEXT NOT NULL,
  ref_id          TEXT,
  idempotency_key TEXT,
  payload         TEXT NOT NULL,
  created_at      TEXT NOT NULL,
  expires_at      TEXT NOT NULL,
  delivered       INTEGER DEFAULT 0
) STRICT;

CREATE INDEX idx_relay_outbox_target ON relay_outbox(target_site_id, delivered)
  WHERE delivered = 0;
```

**`relay_inbox`** — messages received for this host:

```sql
CREATE TABLE relay_inbox (
  id              TEXT PRIMARY KEY,
  source_site_id  TEXT NOT NULL,
  kind            TEXT NOT NULL,
  ref_id          TEXT,
  idempotency_key TEXT,
  payload         TEXT NOT NULL,
  expires_at      TEXT NOT NULL,
  received_at     TEXT NOT NULL,
  processed       INTEGER DEFAULT 0
) STRICT;

CREATE INDEX idx_relay_inbox_unprocessed ON relay_inbox(processed)
  WHERE processed = 0;
```

Inbox inserts use `INSERT OR IGNORE` for natural dedup on UUID PK (handles eager push + sync delivering the same message).

### Message Kinds (This Scope)

Request kinds: `tool_call`, `resource_read`, `prompt_invoke`, `cache_warm`, `cancel`.
Response kinds: `result`, `error`.

The `kind` column is TEXT (no enum constraint) — future plans add `inference`, `stream_chunk`, `stream_end`, `intake`, `process`, `platform_deliver`, `event_broadcast`, `status_forward` without migration.

### Request/Response Payloads

```typescript
// Request payloads (requester -> target)
interface ToolCallPayload {
  tool: string;
  args: Record<string, unknown>;
  timeout_ms: number;
}

interface ResourceReadPayload {
  resource_uri: string;
  timeout_ms: number;
}

interface PromptInvokePayload {
  prompt_name: string;
  prompt_args: Record<string, unknown>;
  timeout_ms: number;
}

interface CacheWarmPayload {
  paths: string[];
  timeout_ms: number;
}

// Response payloads (target -> requester)
interface ResultPayload {
  stdout: string;
  stderr: string;
  exit_code: number;
  execution_ms: number;
}

interface ErrorPayload {
  error: string;
  retriable: boolean;
}
```

Payload size limit: 2MB per message, enforced at outbox insert and inbox insert. Cache-warm responses split multi-file results into one relay message per file (reassembled by `ref_id`).

### Sync RELAY Phase

The sync exchange becomes 4-phase: PUSH → PULL → ACK → RELAY.

**Spoke sends** (step 4a): all undelivered `relay_outbox` entries.
**Hub responds** (step 4b): relay inbox entries for this spoke, `relay_delivered` IDs, `relay_draining` flag.
**Spoke confirms** (step 4c): marks delivered, inserts received via `INSERT OR IGNORE`.

Hub routing on receipt:
- **Target is hub itself:** Execute locally during RELAY phase, return result in step 4b (single round-trip).
- **Target is another spoke:** Store in hub's outbox, deliver on target's next sync. Attempt eager push if reachable.

Wire format additions to sync request:

```typescript
interface SyncRequest {
  push: { events: ChangeLogEntry[] };
  pull_cursor: number;
  relay_outbox: RelayMessage[];  // NEW
}

interface SyncResponse {
  events: ChangeLogEntry[];
  relay_inbox: RelayMessage[];      // NEW
  relay_delivered: string[];         // NEW - outbox IDs hub accepted
  relay_draining: boolean;           // NEW - hub migration drain flag
}
```

### Eager Push

When the hub receives a relay message and the target spoke is reachable, it POSTs immediately to `POST /api/relay-deliver` (Ed25519 authenticated, accepts only from current hub).

Reachability tracking (hub-local, in-memory): per-spoke boolean, initialized `true`. Successful sync/push → `true`. 3 consecutive push failures → `false`. Reset on next successful sync. Not persisted.

### RELAY_WAIT Sub-State

Inside TOOL_EXECUTE, when a tool is remote:

1. Write request to `relay_outbox` with target, kind, payload, idempotency_key, expires_at.
2. Trigger immediate sync cycle.
3. Poll: check `relay_inbox` for `result`/`error` matching `ref_id`. Found → return. Expired → timeout error. Otherwise sleep `sync_interval / 2`, trigger sync, repeat.

Activity status updates to `"relaying {tool_name} via {host_name}"`.

### Tool Routing

Replaces `proxyToolCall()` in `packages/agent/src/mcp-bridge.ts`:

1. Query `hosts` table for hosts listing tool in `mcp_tools`.
2. Filter by recency (exclude `last_sync_at` older than `2 × sync_interval`).
3. Select target: prefer most recent sync, tiebreak on fewer pending outbox entries.
4. Write request → enter RELAY_WAIT.
5. On timeout → try next eligible host. All exhausted → return error.

### Target-Side Execution

Background inbox processor loop on every host (separate from agent loop):

1. **Validate:** Tool/resource/prompt exists locally, requester in keyring.
2. **Idempotency cache check:** In-memory `Map<string, {response, expires_at}>`, 5-minute TTL. Cache hit → return without re-executing.
3. **Expiry check:** Past `expires_at` → discard (requester timed out).
4. **Execute:** Resolve to local MCP client, call tool, capture result.
5. **Write response:** `relay_outbox` entry with `kind = 'result'`/`'error'`, `ref_id` = request ID, `target_site_id` = requester.
6. **Mark processed.** Trigger sync.

### Cancel Handling

When user presses Cancel during RELAY_WAIT:
1. Stop waiting for response.
2. Send `cancel` kind message to target via relay (best-effort, short TTL).
3. Preserve already-persisted tool messages.
4. Add cancellation `system` message.
5. Return to IDLE.

For MCP tool calls, cancel is best-effort — tools can't be interrupted mid-execution. If the tool finishes before cancel arrives, the result is sent back but the requester discards it.

### Idempotency

Request idempotency_key: deterministic hash of `(kind, tool_or_resource, args_hash, timestamp_rounded_to_60s)`. Enforced at:
- **Hub:** Rejects duplicate outbox pushes.
- **Target:** In-memory cache returns cached result without re-execution.

### Hub Migration Drain

`boundctl set-hub` gains a relay drain before switching:

**Phase 1 — DRAIN:** Old hub sets `relay_draining: true` in sync responses. Spokes hold back request-kind entries. Response-kind and `cancel` entries still flow. Hub delivers its own outbox until empty or timeout (120s).

**Phase 2 — SWITCH:** Same as base spec. Spokes' held requests deliver to new hub on first sync.

### Metrics

New columns on `turns`:
- `relay_target TEXT` — target host_name; NULL when local
- `relay_latency_ms INTEGER` — round-trip time; NULL when local

New `relay_cycles` table (non-synced, pruned after 30 days):

```sql
CREATE TABLE relay_cycles (
  id              TEXT PRIMARY KEY,
  ts              TEXT NOT NULL,
  direction       TEXT NOT NULL,       -- 'sent' | 'received'
  peer            TEXT NOT NULL,
  kind            TEXT NOT NULL,
  delivery_method TEXT,                -- 'sync' | 'eager_push' (hub only)
  latency_ms      INTEGER,
  expired         INTEGER DEFAULT 0,
  success         INTEGER NOT NULL
) STRICT;
```

## Existing Patterns

**Change-log outbox pattern** (`packages/core/src/change-log.ts`): The relay outbox/inbox follows the same store-and-forward pattern as the change_log. The change_log writes rows in a transaction, then sync reads them for delivery. Relay tables mirror this: write to outbox, sync delivers, mark delivered. The key difference is relay tables are ephemeral (pruned after delivery) while change_log entries persist for cursor tracking.

**Sync protocol extension** (`packages/sync/src/sync-loop.ts`, `packages/sync/src/routes.ts`): The RELAY phase extends the existing sync loop with a 4th phase. The wire format additions (`relay_outbox`/`relay_inbox` arrays) follow the same JSON structure as the existing `events` array. Hub routing logic lives alongside existing sync route handlers.

**Ed25519 authentication** (`packages/sync/src/signing.ts`, `packages/sync/src/middleware.ts`): The `/api/relay-deliver` endpoint and relay message validation reuse the existing signing infrastructure. Same headers (`X-Site-Id`, `X-Timestamp`, `X-Signature`, `X-Agent-Version`), same verification logic.

**MCP bridge proxy pattern** (`packages/agent/src/mcp-bridge.ts`): The existing `proxyToolCall()` resolves tools via `hosts.mcp_tools`, selects a target by sync recency, and signs HTTP requests. Relay routing follows the same discovery and selection logic but writes to the outbox instead of making a direct HTTP call. `generateMCPCommands()` handler changes from direct HTTP to relay write.

**Schema migrations** (`packages/core/src/schema.ts`): New tables follow the same `CREATE TABLE ... STRICT` pattern. Non-replicated tables (like `change_log`, `sync_state`, `host_meta`) are documented in the non-replicated tables list.

**No new patterns introduced.** The relay is a natural extension of existing sync, outbox, and MCP bridge patterns.

## Additional Considerations

**Payload size for cache-warm:** When cache-warm responses contain multiple files exceeding 2MB total, the target splits into one relay message per file, each under 2MB. The requester reassembles by `ref_id`. Individual files exceeding 2MB are rejected (consistent with the overlay file size limit of 1MB per the base spec §4.2).

**Quiescence interaction:** Reduced sync frequency during quiescence increases relay latency proportionally. This is benign — quiescence activates when no user has interacted in hours, meaning no interactive relay calls are in flight. Activity lifts quiescence and latency normalizes.

**Future extensibility:** The `kind` TEXT column and `RelayMessage` type are designed to accept new kinds without migration. The streaming kinds (`inference`, `stream_chunk`, `stream_end`), platform kinds (`intake`, `process`, `platform_deliver`), and event kinds (`event_broadcast`, `status_forward`) will be added in separate design plans. The `stream_id` column from the RFC is omitted in this scope (no streaming) and will be added via migration when the inference relay plan is implemented.

**Confirm gates:** Checked BEFORE writing to relay outbox, identical to current behavior. No change in confirm gate semantics — if user declines, no relay message is sent.
